## What The App Is

- Local-first planning app built with Rust and Dioxus
- Single workspace card at the top of the tree
- Every other board is a normal child card
- Cards show their immediate children as compact previews
- Notes and due dates are built in
- Browser persistence uses localStorage
- Native builds currently fall back to in-memory state

## Current Architecture

### Domain

- `CardId` and feature id newtypes in `src/domain/id.rs`
- `Card` in `src/domain/card.rs`
- `NotePage` and `DueDate` in `src/domain/`
- `DomainError` in `src/domain/error.rs`
- `CardRegistry` in `src/domain/registry.rs`

Core model:

- `Card` has `id`, `title`, `parent_id`, `children_ids`, `notes`, and `due_date`
- No buckets remain in the model
- Workspace is the single top-level card
- Child ordering is preserved through `children_ids`

### Application

- `Command` dispatcher in `src/application/mod.rs`
- `BoardView` projection for a board's ordered children
- `CardPreviewView` projection for a card's immediate children
- command execution owns command lifecycle logging

### Infrastructure

- `src/infrastructure/repository.rs` handles persistence
- Browser `localStorage` is the active persistence path
- Legacy bucket-shaped payloads are treated as incompatible and reset to a clean workspace

### Interface

- `src/interface/app.rs` wires app state and routing
- `src/interface/routes/home.rs` renders the workspace card
- `src/interface/routes/board.rs` renders child boards
- `src/interface/components/board_view.rs` renders the wrapping child-card grid
- `src/interface/components/card_item.rs` renders a card plus its compact immediate-child preview
- `src/interface/components/layout.rs` renders the top bar and web utilities
- `src/interface/components/modal/` holds the feature-focused modals

## Current UI Behavior

- Workspace and child boards use the same card chrome
- Back button is disabled on the workspace card because there is no higher level
- Create Card, Notes, and Settings live in the board header
- Child cards render in a wrapping grid, not horizontal columns
- Drag/drop is card-only
- Inline child previews are shallow and only show one level down

## Learning Notes

- `Result<T, DomainError>` is the main validation pattern
- `Option<T>` represents optional parent, due date, or modal state
- Dioxus `Signal<T>` holds shared reactive state
- `rsx!` is the UI declaration syntax
- `#[cfg(target_arch = "wasm32")]` separates browser-only behavior

## Glossary

- `Card` - the main planning item and board owner
- `CardRegistry` - workspace-wide state container
- `BoardView` - ordered child-card projection for a board
- `CardPreviewView` - immediate-child preview projection for a card
- `Command` - mutation request handled by the application layer
- `ModalType` - UI state describing which modal is open
- `DomainError` - typed validation or invariant failure

## What Changed Recently

- Buckets were removed entirely
- The workspace became the single top-level card
- Old bucket-based snapshots are reset on load instead of migrated
- The UI now uses a pure card hierarchy
- Card previews show only immediate children
- The smoke test was updated to the card-only flow

## Short Model Summary

The app works like this:

1. Load the workspace card.
2. Render its ordered child cards in a wrapping board.
3. Let each card open into its own board.
4. Show immediate child previews inside cards.
5. Persist the registry to browser storage when on web.

That is the current mental model for the entire application.
